# MasterBot

Toolkit for Master Vacation Homes customer service agents. Three pieces that
share one Supabase project:

```
 ┌─────────────────┐     ┌─────────────────────┐     ┌──────────────────┐
 │  Site (Pages)   │     │  Chrome Extension   │     │     Supabase     │
 │  alissonrochah  │◄───►│    Resort Info      │◄───►│   (auth + DB)    │
 │    .github.io   │ ext │  (Airbnb inbox)     │     │                  │
 └─────────────────┘     └─────────────────────┘     └──────────────────┘
```

- **Site** — web app for writing messages (templates + AI) and managing resorts.
- **Extension** — injected into the Airbnb hosting inbox. Auto-matches the
  listing title against the resort list and drops a "Gate Code" button next
  to Airbnb's suggested door code button.
- **Supabase** — one project shared by site and extension (auth, resorts,
  templates, user settings, account management).

## How the pieces talk

**Site → Supabase.** Every page imports `js/supabase.js` (which pulls
`@supabase/supabase-js` from esm.sh) and uses the usual Supabase SDK for
auth, RLS-protected reads, and admin RPC calls.

**Extension → Supabase.** The extension can't import from a CDN under
Manifest V3's CSP, so it has its own minimal REST client in
`extension/js/api.js`. It talks to the same URL and uses the same anon key.
Sessions live in `chrome.storage.local`.

**Extension ↔ Site.** The AI page (`ai.html`) talks to the extension via
`chrome.runtime.sendMessage(EXTENSION_ID, ...)`. Two actions:

- `getMatchedResort` — returns the resort the background worker matched
  against the currently-open Airbnb listing title.
- `scrapeConversation` — the background worker relays this to the content
  script on the active Airbnb tab, which scrapes the message thread and
  returns the parsed messages.

The extension ID is stored per-user in `user_settings.extension_id` so
different developers/machines can use their own builds.

**Extension ↔ Airbnb tab.** The content script watches Airbnb's
`#thread_details_panel`, extracts the listing title, and writes it to
`chrome.storage.local`. The background worker matches that title against
the cached resort list and publishes the matched resort back to storage.
The content script reads the match and injects a "Gate Code" button next
to the existing "Suggested door code" button.

## Running it

**Site.** GitHub Pages serves the repo root. Any push to `main` deploys.
There is no build step.

**Extension.**
1. Open `chrome://extensions`, enable Developer Mode.
2. Click "Load unpacked" and select the `extension/` folder.
3. Copy the extension ID and paste it into your account's Settings page
   (`settings.html`) so the site can talk to your specific build.

**Database.** Run the SQL files under `db/` in the Supabase SQL editor in
order: `schema.sql`, `schema_accounts.sql`, `schema_templates.sql`. The
seed file is one-off and only re-run when the source xlsx changes.

**Regenerating the seed.**
```bash
cd tools
python -m venv .venv && source .venv/bin/activate
pip install openpyxl
python build_seed.py
```
This rewrites `db/seed_all_resorts.sql` in place.

## Supabase tables (what each page reads)

| Table | Read/Written by |
|---|---|
| `resorts` | site/dashboard.js, site/ai.js, extension/popup.js |
| `templates` | site/messages.html, site/template.html, site/ai.html |
| `profiles` | site/accounts.js, site/resorts/index.html |
| `user_settings` | site/settings.html, site/accounts.js |
| `global_settings` | site/settings.html, site/ai.js (Groq key lookup) |

All write-paths are guarded by RLS. Account creation/deletion goes through
`admin_create_account` / `admin_update_account` / `admin_delete_account`
RPCs (SECURITY DEFINER) which atomically maintain `auth.users`,
`auth.identities`, and `profiles`.
